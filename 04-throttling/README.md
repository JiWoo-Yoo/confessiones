# 04. 쓰로틀링, 디바운싱 그리고 배칭 삼총사

[playground 페이지 확인](https://confessiones.vercel.app/)

## 📌 Issue Overview

- 실무에서 디바운싱은 알고 있었으나, 쓰로틀링을 몰라 타 개발자가 쓰로틀링을 해야한다고 했을 때 '예? 그게 뭐죠?' 라고 했던 치욕에 대한 회고 및 보충.
- 무분별한 이벤트 발생(Scroll, Input)으로 인한 브라우저 성능 저하 및 불필요한 API 호출 방지.

## 🔍 개념 및 사용 용도 확인

### 1. 쓰로틀링 (Throttling) - "적당한 리듬으로 끊기"

- 핵심: "아무리 많이 요청해도, 정해진 시간 간격(Interval)마다 최대 한 번만 실행한다."

- 해결: 실행 중이면 다음 요청을 무시하고, 설정한 시간(예: 300ms)마다 주기적으로 딱 한 번씩만 실행되게 함. 흐름을 유지하며 성능을 잡는 게 핵심.

- 용도: 무한 스크롤, 스크롤 애니메이션, 마우스 이동 추적.

### 2. 디바운싱 (Debouncing) - "다 말할 때까지 기다리기"

- 핵심: 마지막 요청이 끝난 후에만 실행한다. 유저가 검색창에 '안녕'이라고 칠 때 'ㅇ,아,안,안ㄴ,안녀,안녕'마다 API를 쏘는 건 서버 학대임.

- 해결: 연달아 들어오는 신호는 무시하고, 마지막 입력이 끝난 후 일정 시간(예: 500ms) 동안 추가 입력이 없을 때만 최종적으로 한 번 실행.

- 용도: 검색창 자동완성, 윈도우 리사이즈(Resize) 대응.

3. 배칭 (Batching) - "택배는 모아서 한 상자에"

- 핵심: "내가 직접 구현하는 게 아니라, 프레임워크(React 등)가 알아서 묶어서 처리해주는 자동 최적화."

- 리액트에서 setState를 3번 연속 쓰면 화면을 3번 다시 그려야 할 것 같지만, 리액트는 영리하게 이걸 하나로 묶음.

- 여러 상태 변경을 단 한 번의 리렌더링으로 처리하여 브라우저의 연산 횟수를 획기적으로 줄임. (React 18의 Automatic Batching이 대표적)
